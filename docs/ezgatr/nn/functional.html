230    r"""Perform Pin-equivariant linear map defined by weight on input x.
231
232    One way to think of the equivariant linear map is a channel-wise
233    "map-reduce", where the same weight (of one neuron) are applied to
234    all channels of the input multi-vector and the results are summed up
235    along the basis/blade axis. In other words, the map is a channel-mixing
236    operation. Using a parallel analogy with a regular ``nn.Linear`` layer,
237    each channel of a input multi-vector corresponds to a "feature value"
238    of a simple hidden representation, and the number of output channels
239    is the number of neurons in the hidden layer.
240
241    Within each channel, similar to the geometric product implementation,
242    the linear operation starts with a matrix multiplication between the
243    (weighted, if normalized) 16-by-16 computation graph, i.e., the basis,
244    and the input multi-vector to take into account the effect that blades

247    to optimize this operation as the basis is pretty sparse.
248

253        x and weight.

256        broadcastable between x and weight.

259        scalar blades (i.e., index position 0) of each output channel.

272    #   o: output channels

274    #   w: learnable parameters
275    #   d: destination blades
276    #   s: source blades
277    x = torch.einsum("oiw, wds, ...is -> ...od", weight, basis, x)
278    if bias is not None:
279        x[..., [0]] += bias.view(-1, 1)
280    return x

One way to think of the equivariant linear map is a channel-wise "map-reduce", where the same weight (of one neuron) are applied to all channels of the input multi-vector and the results are summed up along the basis/blade axis. In other words, the map is a channel-mixing operation. Using a parallel analogy with a regular nn.Linear layer, each channel of a input multi-vector corresponds to a "feature value" of a simple hidden representation, and the number of output channels is the number of neurons in the hidden layer.

Within each channel, similar to the geometric product implementation, the linear operation starts with a matrix multiplication between the (weighted, if normalized) 16-by-16 computation graph, i.e., the basis, and the input multi-vector to take into account the effect that blades containing e_0 will be "downgraded" to a lower grade after the map. Again, a map from "source-to-destination" style. Note that we may want to optimize this operation as the basis is pretty sparse.

10    x: torch.Tensor, weight: torch.Tensor | None = None, eps: float | None = None
11) -> torch.Tensor:
12    r"""Compute PGA inner-induced RMS norm of multi-vectors.
13
14    Although the original GATr paper [2]_ named the normalization operation
15    as E(3)-equivariant LayerNorm, we find the actual implementation more
16    similar to RMSNorm [1]_ by substituting the scalar squaring with the inner
17    product of multi-vectors, i.e., PGA inner-induced RMSNorm.
18
19    We find the adaptation more intuitive by viewing each channel of the input
20    multi-vector as an "imaginary number", so that the inner product of the
21    multi-vector with itself is the squared "modulus". And, in turn, instead of
22    thinking the input tensor as channels-of-multi-vectors, we can think of it
23    as feature-vector-of-imaginary-numbers. And the normalization operation is
24    the same as the RMSNorm operation on the scalar-valued feature vectors.
25
26    NOTE: THIS DESIGN IS PROBABLY QUESTIONABLE. The inner product do not
27    consider coefficients associated with blades containing ``e_0`` while
28    the scaling is also applied to these blades.
29
30    Parameters
31    ----------
32    x : torch.Tensor
33        Input multi-vectors with shape (..., n_channels, 16).
34    weight : torch.Tensor, optional
35        weight for re-scaling the normalized input. It can be both
36        static or learnable, depending on how ``weight`` are initialized
37        outside of the function.
38    eps : float, optional
39        Small value to prevent division by zero.
40
41    Returns
42    -------
43    torch.Tensor
44        Normalized input with shape (..., n_channels, 16).
45
46    Reference
47    ---------
48    .. [1] `"Root Mean Square Layer Normalization", Biao Zhang and Rico Sennrich,
49            <https://ar5iv.labs.arxiv.org/html/1910.07467>`_
50    .. [2] `"Geometric Algebra Transformer", Johann Brehmer, et al.,

52    """

54    norm = torch.mean(inner_product(x, x), dim=-2, keepdim=True)

We find the adaptation more intuitive by viewing each channel of the input multi-vector as an "imaginary number", so that the inner product of the multi-vector with itself is the squared "modulus". And, in turn, instead of thinking the input tensor as channels-of-multi-vectors, we can think of it as feature-vector-of-imaginary-numbers. And the normalization operation is the same as the RMSNorm operation on the scalar-valued feature vectors.
